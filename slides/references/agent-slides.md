## Slide Structure Template A — Standard Ablation Study

```
1.  Title slide          — Experiment name, design matrix summary (N papers x M models x K prompts)
2.  Motivation           — Why this experiment, three objectives
3.  Evaluation criteria  — Exact definition of each verdict/metric + key distinctions (e.g. WARN vs FAIL)
4.  Experiment design    — Full matrix card
── Ablation explanation zone (one slide per variable) ──
5.  Dataset / papers     — One row per subject: ID, title, domain, special traits
6.  Model comparison     — Capability dimension table (reasoning depth, accuracy, bias)
7.  Prompt version diff  — Line-by-line actual differences, expected vs actual effect
── Results zone ──
8.  Behavior profiles    — PASS/FAIL/WARN breakdown per config (with animated bar charts)
9.  Distribution overview — Stacked bar chart (all combinations side by side)
10. Heatmap (key finding) — Most prominent paper x section matrix
── Deep analysis zone ──
11. Case 1               — Real high-quality bug found, with model verdict gradient
12. Case 2               — Representative case of another category
13. Section risk ranking — Horizontal bar chart (sorted by FAIL count)
14. Variable effect comparison — Numeric comparison table + three findings per variable
── Summary ──
15. Core conclusions     — 4 findings, each with data support
```

**Core principle:** Every ablation element needs at least one explanation slide appearing *before* its results slide. The audience should not see "X's performance" before knowing "what X is."

## Content Source Rules

| Content type | Source |
|-------------|--------|
| Model behavior data | `summary.md` generated by `analyze.py` |
| Case study details | Read directly from `.review/{task_id}/integrity_audit.md` |
| Paper descriptions | First 10 lines of abstract from `paper_grobid.md` |
| Prompt differences | Compare two version files under `agents/prompts/sys_*/` |

**Do not reconstruct data from memory.** Case study details must come from actual audit reports.

## Visualization Selection

| Scenario | Use |
|----------|-----|
| Distribution of N configs under one variable | Stacked horizontal bar chart (PASS/FAIL/WARN segments) |
| Paper x Model cross matrix | Heatmap table (colored verdict badges) |
| All sections of a single paper | Row-highlighted heatmap (accent background on highlighted rows) |
| Single-variable ranking | Monochrome horizontal bar chart (sorted by value) |
| Config capability comparison | Plain HTML table (rows = dimensions, columns = configs) |
| Findings / conclusions | Finding Card (left colored border: red=critical, yellow=warning, blue=info) |

## Visual Style Spec — GitHub Dark (production standard)

For technical internal reports, use GitHub Dark theme:

```css
--bg: #0d1117;         /* main background */
--bg2: #161b22;        /* card / panel background */
--bg3: #21262d;        /* nested element background */
--border: #30363d;     /* borders */
--text: #e6edf3;       /* body text */
--muted: #8b949e;      /* secondary text */
--accent: #58a6ff;     /* highlight blue */
--accent2: #79c0ff;    /* secondary highlight */
--pass: #3fb950;       --fail: #f85149;
--warn: #e3b341;       --pending: #8b949e;
```

**Fonts:** `Inter` (body) + `JetBrains Mono` (data / code / labels)

**All sizes use `clamp()` for responsive scaling — no media query breakpoints:**
```css
--title-size: clamp(2rem, 5.5vw, 4rem);
--body-size:  clamp(0.75rem, 1.3vw, 1rem);
--small-size: clamp(0.6rem, 0.95vw, 0.8rem);
```

**Verdict badge CSS:**
```css
.badge-pass    { color: var(--pass);    background: rgba(63,185,80,.15); }
.badge-fail    { color: var(--fail);    background: rgba(248,81,73,.15); font-weight: 700; }
.badge-warn    { color: var(--warn);    background: rgba(227,179,65,.15); }
```

## Template Selection Guide

### Template A — Standard Ablation Study (15 slides)

Use when: comparing multiple models x prompts x papers to find which combination performs best.

Focus: "Which configuration is best?"

### Template B — Attack Recall Benchmark

Use when: fixed best configuration, injecting defects into multiple papers to evaluate recall.

```
1. Title slide          — Experiment name, config summary (N papers x 1 config)
2. Experiment design    — Three-phase pipeline (Attack → Review → Evaluate)
3. Per-Paper results    — Per-paper recall/FPR table + inline bar charts
4. Category distribution — Detection rate per defect category + blind spots
5-6. Case Study x2      — Injected defect vs audit detection (inject/detect layout)
7. FP analysis          — False positive breakdown + FPR comparison with other experiments
8. Conclusions          — Finding Cards grid + Next Steps
```

Focus: "Which papers / categories are detected well?"

**Pipeline visualization component:**
```css
.pipeline {
  display: grid;
  grid-template-columns: 1fr auto 1fr auto 1fr;
  align-items: start;
}
.pipeline-arrow { color: var(--muted); padding: 0 .75rem; }
.pipeline-col {
  background: var(--bg2);
  border: 1px solid var(--border);
  border-radius: 10px;
  padding: 1rem;
}
```

**Bug layout (inject vs detect):**
```css
.bug-layout {
  display: grid;
  grid-template-columns: 1fr 1.4fr;
  gap: 1.5rem;
}
.ba-card.injected { border-color: var(--fail); background: rgba(248,81,73,.05); }
.ba-card.detected { border-color: var(--pass); background: rgba(63,185,80,.05); }
```

### Template C — Multi-Factor Ablation

Use when: N configuration combos (model x prompt) x M papers, evaluating recall/FPR across all.

```
1.  Title slide         — Experiment name, design matrix (N combos x M papers)
2.  Experiment design   — Pipeline + matrix card
3.  Glossary            — Glossary slide (core metrics / experiment design / audit rules)
── Results zone ──
4.  Overview heatmap    — Combo x Paper recall/FPR matrix (colored badges)
5.  Variable effects    — Comparison table split by single factor (model / prompt)
6.  Category distribution — Detection rate comparison per defect category
── Deep analysis zone ──
7-8. Case Study x2      — inject/detect comparison + verdict diff per combo
9.  FP analysis         — False positive breakdown + cross-combo comparison
── Summary ──
10. Conclusions         — Finding Cards + recommended configuration
```

Focus: "Which combination is best?" (vs Template B which focuses on which papers/categories)

## Theme Consistency Rule

**All experiment slides within the same project must use the same visual theme.**

If old slides use a different theme (e.g., Tailwind color palette, Space Grotesk font), migrate CSS variables to the standard GitHub Dark theme rather than rebuilding the entire file.

Migration steps:
1. Replace all color variables in `:root` with GitHub Dark tokens
2. Replace font declarations with Inter + JetBrains Mono
3. Update verdict badge colors and backgrounds
4. Check nav dots, progress bar, grid background, and other decorative elements

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Show model results before explaining what the model is | Ablation explanation slide must come before the corresponding results slide |
| Case study restated from summary, no concrete evidence | Read audit report directly, cite specific line numbers / values |
| All charts are bar charts | Cross matrix uses heatmap, single-factor ranking uses horizontal bar |
| Conclusion slide is text only | Each conclusion must include data ("89 FAILs vs 13 FAILs") |
| Forgot viewport fitting | Every slide: `height: 100vh; overflow: hidden;`, all text uses `clamp()` |
| Domain terms used without explanation | When Recall/FPR/pp/Ablation appear, must add Glossary slide |
| Multiple experiment slides with inconsistent themes | Unify all slides in project to GitHub Dark, migrate CSS variables for old slides |
| Slide IDs hard-coded then renumbering forgotten | After inserting new slide, check all `id="slide-N"`, counters, and nav links |
